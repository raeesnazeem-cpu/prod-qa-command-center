Cloudinary Image Storage Plan

Goal: Migrate thumbnail and screenshot storage to a reliable, free image hosting service (Cloudinary) and ensure images are automatically deleted when their parent "Runs" are deleted.

1. Why are current Supabase images "disappearing"?

Currently, depending on the bucket configuration (e.g., private buckets), the system might be generating Signed URLs that expire after a certain time (e.g., 1 hour). When you check the database later, the URL is broken, making it look like the image was deleted. Some jobs use public buckets, but the private ones will definitely exhibit this "disappearing" behavior.

2. Is Cloudinary the right choice?

Yes, absolutely. Cloudinary is the industry standard for this exact use case.

Feature Cloudinary Free Tier
Storage 1 GB (or 25 Credits total where 1 credit = 1GB or 1k transformations)
Transformations Automatic thumbnail sizing and compression on the fly via URL parameters
Permanence URLs never expire unless you explicitly delete the image

3. Proposed Architecture & Workflow

A. Uploading Images

  1. Update the uploadScreenshot logic in the Worker and API.
  2. Instead of uploading to Supabase, we upload the buffer directly to Cloudinary.
  3. Crucial Step: We will tag every uploaded image with a tag matching the Run ID (e.g., run_12345). This makes bulk deletion extremely easy.
  4. Cloudinary returns a permanent secure URL (e.g., https://res.cloudinary.com/...) which we save to the Supabase database.

B. Deleting Images (Cleanup)

We need to ensure images don't accumulate when runs are deleted. There are two ways to handle this:

  • Option 1 (API Triggered): When a DELETE request is made to /api/runs/:id, before deleting the run from Supabase, we call the Cloudinary API: cloudinary.api.delete_resources_by_tag("run_12345"). This instantly wipes all screenshots and thumbnails associated with that run.
  • Option 2 (Supabase Webhook): Create a database webhook in Supabase that listens for DELETE events on the runs table and triggers an Edge Function to delete the Cloudinary resources. (Option 1 is simpler and recommended for now).

4. Implementation Steps

  1. Account Setup: Create a free Cloudinary account and get the CLOUD_NAME, API_KEY, and API_SECRET. Add these to the .env file.
  2. Install SDK: Run pnpm add cloudinary in the worker and api packages.
  3. Create Cloudinary Helper: Create lib/cloudinaryStorage.ts containing the upload and delete logic.
  4. Update Workers & API: Modify captureScreenshotJob, captureMultiviewScreenshotsJob, and routes to use the new uploader and pass the run_id as a tag.
  5. Update Delete Logic: Modify the Run deletion logic in the API to call the Cloudinary deletion helper using the tag.

5. Example Code Sneak Peek

// lib/cloudinaryStorage.ts
import { v2 as cloudinary } from 'cloudinary';

cloudinary.config({ 
  cloud_name: process.env.CLOUDINARY_CLOUD_NAME, 
  api_key: process.env.CLOUDINARY_API_KEY, 
  api_secret: process.env.CLOUDINARY_API_SECRET 
});

export async function uploadToCloudinary(buffer: Buffer, runId: string) {
  return new Promise((resolve, reject) => {
    const uploadStream = cloudinary.uploader.upload_stream(
      { tags: [\`run_\${runId}\`], folder: 'qa-command-center' },
      (error, result) => {
        if (result) resolve(result.secure_url);
        else reject(error);
      }
    );
    uploadStream.end(buffer);
  });
}

export async function deleteRunImages(runId: string) {
  // Deletes all images tagged with this run ID instantly
  await cloudinary.api.delete_resources_by_tag(\`run_\${runId}\`);
}

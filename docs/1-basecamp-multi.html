Multi-User Basecamp Sign-In Implementation Plan

Hello! This document explains how we will build the Multi-User Basecamp Sign-In feature. The goal is to allow every regular user to push to Basecamp using their very own profile.

Important Requirement: This change will be purely additive. We will not break any existing code. If the user is a superadmin, the system will gracefully fall back to using the common project token. For everyone else, it will enforce the use of a personal token.

Part 1: Database Changes (Backend Storage)

We are only adding new things, not deleting anything. Since we already have the basecamp_person_id for each user in our system, half of our job is already done! We just need a place to securely store their actual login keys.

Step 1: Add new token columns to the `users` table

We will add the following columns to the Supabase users table:

  • basecamp_access_token (Text): The main key used to push messages on behalf of the user.
  • basecamp_refresh_token (Text): A backup key used to get a new main key when the main one expires.
  • basecamp_token_expires_at (Timestamp): To check if the main key is expired and needs refreshing.

In simple terms: We are just creating new blank boxes in our database row for each user's secret keys. No existing data is touched.

Part 2: The Basecamp Login Flow (Backend APIs)

Basecamp uses a system called OAuth2. A user clicks a button, goes to the Basecamp website, clicks "Allow", and Basecamp sends a secret code back to us.

Step 2: Create a "Start Login" API route (New Code)

We will make a new backend link: GET /api/basecamp/user-auth.

When the user clicks "Connect Basecamp" on our frontend, it hits this API, which redirects their browser to the Basecamp login page.

Step 3: Create a "Callback" API route (New Code)

After the user says "Allow", they come back to a special link like GET /api/basecamp/callback.

Our backend takes the secret code from the URL, talks to Basecamp secretly, and exchanges it for the user's basecamp_access_token. We then save this token into the users table.

Step 4: Make a Token Refresher (New Code)

Basecamp tokens expire after a while. We will write a small helper function in the backend that automatically fetches a fresh token using the basecamp_refresh_token just before doing any push.

Part 3: Additive Push Logic Update (Backend APIs)

We are going to carefully wrap the existing push logic in a new if/else condition.

Step 5: Safely change the push route (`POST /:id/basecamp`)

When someone tries to push to Basecamp, the backend will do the following check:

  1. Check the User's Role: Is this person a superadmin?
  2. Check the User's Token: Does this person have a personal basecamp_access_token saved in their profile?

The Logic rules:

  • If they have a personal token: Use their personal token to push the task.
  • If they DO NOT have a personal token AND they are a superadmin: Fall back to the old logic. Use the common projectSettings.basecamp_token.
  • If they DO NOT have a personal token AND they are NOT a superadmin: Stop the process. Send an error back saying "Please connect your personal Basecamp account to push tasks."

This guarantees we don't break anything for superadmins, but we strictly force everyone else to connect.

Part 4: Frontend UI Changes (HTML/CSS & React side)

This is what the user will actually see on the screen.

Step 6: Update Existing User Settings Page

Since we already have a SettingsPage per user, we will use it to store and manage these user settings.

Add a "Connect to Basecamp" button inside this existing page. If they are already connected, show a green tick ✅ "Connected" with a "Disconnect" option.

Step 7: Check before pushing (Additive UI Update)

On the page where they click "Push to Basecamp", we check if they are allowed.

  • For Superadmins: Show the push button normally, always.
  • For Regular Users:
    • If connected: Show the push button.
    • If NOT connected: Show a popup modal saying, "Kindly connect your Basecamp account first before pushing," with a button that takes them to their Settings page.

Summary of the Effort Required

  • Database Additions: ~1 hour
  • Basecamp OAuth Login Flow: ~4-5 hours
  • Additive Push Logic Updates: ~2-3 hours
  • Frontend UI: ~3-4 hours

Total Estimated Time: About 2 full working days (10 to 14 hours of development and testing).

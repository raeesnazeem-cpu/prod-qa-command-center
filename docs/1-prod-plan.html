Plan B: Production Deployment Guide

A step-by-step granular guide to push your React/Node app to a live Contabo Server using Dokploy (or aaPanel).

Hello brother! Since you are coming from a basic HTML/CSS background, don't worry at all. I have broken down the entire complex backend deployment into very simple, copy-paste steps.

Basically, what we are doing is shifting our code from localhost to a live internet server so everyone can use it. We will use a Contabo VPS (which is like a blank computer kept in Germany/US) and we will install a Control Panel called Dokploy on it. This panel will give us a very nice UI, so we don't have to deal with scary black screens too much! Let's start.

Phase 0: Granular Level Preparation (Very Important)

Before we even touch the Contabo server, we need to gather all our API keys and setup our 3rd party accounts. Open a simple Notepad on your laptop and copy-paste these keys as we generate them.

A

Clerk (Login & Signup System)

We use Clerk so we don't have to code the login/password security ourselves. It handles all the headache of sessions and tokens.

  • Log in to your existing clerk.com account where the QA Command Center application is currently set up.
  • Go to the API Keys section for your application.
  • Copy the Live NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY and the CLERK_SECRET_KEY. Paste them in your Notepad.
  • Crucial Note: Once your app is live on a real domain (like yourwebsite.com), you MUST come back to Clerk, go to "Domains", and add your new live production domain. This tells Clerk to accept logins from your actual website instead of just localhost.
  • Make sure to update any Redirect URLs in Clerk settings to point to your new production domain.
B

Supabase (Your Database)

This is where all the scan results, projects, and user data are stored forever. It's basically a free online PostgreSQL database.

  • Log in to your existing supabase.com account and open your current qa-command-db project.
  • Since we are migrating to production, we will continue using this existing database to keep all your current data intact.
  • Go to the Settings (Gear icon at the bottom left) -> API section.
  • Copy the Project URL and the anon public key. Put them in your Notepad.
  • Next, go to Settings -> Database. Scroll down to "Connection String (URI)" and copy the string.
  • Replace the password placeholder with your actual existing database password. Put this final connection string in your Notepad.
  • If you need to make any production-specific schema changes, run your existing migrations against this database.
C

Upstash Redis (The Task Queue)

When someone clicks "Scan", we can't do it instantly if 100 people click at the same time. We need a Queue (a waiting line). Redis acts like a super-fast manager handling this waiting line.

  • Log in to your existing upstash.com account and open your current qa-queue database.
  • We will reuse this existing Redis queue for production as well.
  • Scroll down to the REST API section and find the UPSTASH_REDIS_REST_URL and UPSTASH_REDIS_REST_TOKEN.
  • Copy and put them in your Notepad.
D

Google Cloud Platform (GCP - For Video Recording)

Our background worker uses Google Cloud Run to open a fake browser, record the video, and save it to Google Cloud Storage. We need permission keys for this.

  • Go to console.cloud.google.com and select your existing qa-video-recorder project.
  • Since everything is already set up, we just need to retrieve your credentials.
  • If you already have your GCP_SERVICE_ACCOUNT_JSON saved locally, simply copy it to your Notepad.
  • If you lost it: go to IAM & Admin -> Service Accounts, click your existing `video-bot` service account, go to the Keys tab, click "Add Key" -> "Create new key" -> Choose JSON.
  • Open the downloaded JSON file, copy all the text inside, and put it in your Notepad. This block is your GCP_SERVICE_ACCOUNT_JSON.
E

Preparing the Final Environment Variables (.env)

Now, combine all the things from your Notepad into your final `.env` file structure. This text block is EXACTLY what you will copy-paste into Dokploy later for both the Web and the Worker apps.

# 1. Clerk Auth Keys NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY="pk_live_..." CLERK_SECRET_KEY="sk_live_..." # 2. Supabase DB Keys NEXT_PUBLIC_SUPABASE_URL="https://xyz.supabase.co" NEXT_PUBLIC_SUPABASE_ANON_KEY="eyJhbG..." DATABASE_URL="postgresql://postgres.[xyz]:[password]@aws-0.../postgres" # 3. Upstash Redis Queue UPSTASH_REDIS_REST_URL="https://...upstash.io" UPSTASH_REDIS_REST_TOKEN="AYZ..." # 4. Google Cloud (Paste the exact full JSON string inside single quotes) GCP_PROJECT_ID="qa-video-recorder-123" GCP_SERVICE_ACCOUNT_JSON='{ "type": "service_account", "project_id": "..." }' # 5. Basecamp (If you are using the Basecamp integration) BASECAMP_CLIENT_ID="..." BASECAMP_CLIENT_SECRET="..."
Excellent! Now that your Notepad is ready with all the live production keys, we can move on to actually buying the server.

Phase 1: Server Setup & Deployment

Now we will put the code on the Contabo server using Dokploy.

1

Purchasing the Contabo Server (VPS)

First of all, we need to rent the computer.

  • Go to Contabo.com and go to the "Cloud VPS" section.
  • Select the basic "VPS 1" or "VPS 2" plan. It is very cheap and has plenty of RAM (6GB/8GB) for our background worker and web app.
  • Most Important: During checkout, it will ask for the "Operating System" (OS). You MUST select Ubuntu 22.04 or Ubuntu 24.04. Please do not select Windows.
  • Make the payment. Within a few hours, Contabo will send you a welcome email.
  • In that email, you will get an IP Address (something like 123.45.67.89) and a Root Password. Keep this safe!
2

Connecting to your Server (SSH)

Now we need to login to that blank Contabo computer from your Mac.

  • Open the Terminal app on your Mac.
  • Type the following command (replace with your actual IP from the email):
ssh root@123.45.67.89
  • It will show a message asking "Are you sure you want to continue connecting?". Type yes and press Enter.
  • Then it will ask for the password. Note: When you type the password, the screen will NOT show any stars or dots. It will look like nothing is typing. Just type it carefully and press Enter.
  • Congratulations! You are now inside your Contabo server.
3

Installing the Control Panel (Dokploy)

You mentioned "aa panel on dokploy". Both are control panels, but for React/Node.js apps, Dokploy is currently the absolute best. It works exactly like Vercel but on your own cheap server. If you strictly want aaPanel, that is also fine, but Dokploy is highly recommended here for zero-headache React hosting.

To install Dokploy, simply copy and paste this one single magic command in your terminal and press Enter:

curl -sSL https://dokploy.com/install.sh | sh
Now, go drink a cup of chai ☕. This will automatically download and install everything needed (Docker, Nginx, etc.). It takes about 5 to 10 minutes.
  • Once it finishes, it will give you a link in the terminal. It will look like this: http://123.45.67.89:3000
  • Open that link in your Chrome browser.
  • It will ask you to create an Admin Username and Password. Set it up. Now you have a beautiful dashboard!
4

Connecting your GitHub

Your code is currently on your local Mac. Dokploy needs to fetch it.

  • Make sure you have pushed all your latest code to a private repository on GitHub.
  • Inside your new Dokploy browser dashboard, go to Settings -> GitHub.
  • Click "Connect GitHub". This will allow your server to securely download your code without needing passwords every time.
5

Deploying the Frontend (Web App)

Now we will make the actual React website live.

  • In Dokploy, click on Create Application (or Project).
  • Name it: qa-command-web
  • Source: Select your GitHub repository.
  • Build Command: Since it is a monorepo, you need to tell it how to build the web part. Type:
    pnpm install && pnpm build:web
  • Start Command: How to turn it on.
    pnpm start:web
  • Environment Variables: Open your local .env or .env.local file on your Mac. Copy all the text inside it, and paste it into the Environment Variables section in Dokploy.
  • Click Deploy. Dokploy will now install everything, build the React app, and make it live!
6

Deploying the Background Worker

Our app has two parts. The website, and the invisible background worker that checks the AI videos and tasks. We need to run that too.

  • Go back to the Dokploy dashboard and click Create Application again.
  • Name it: qa-command-worker
  • Source: Select the exact same GitHub repository.
  • Build Command:
    pnpm install && pnpm build:worker
  • Start Command:
    pnpm start:worker
  • Environment Variables: Paste the exact same .env file contents here too. (Make sure your Upstash Redis and Supabase URLs are correct here, because the worker needs the database!)
  • Click Deploy. Now your worker is running 24x7 in the background doing its job!
7

External Cloud Services Check

Just to remind you, to keep things simple and cheap, we are not hosting the heavy Database on Contabo. We are using external free tiers.

  • Supabase (Database): All your user data and rows are stored here. Double check that the Supabase keys are present in your Dokploy Environment variables.
  • Upstash Redis (Queue): The worker uses this to line up tasks. Double check the Redis URL is correct.
  • Google Cloud (GCP): For the video recording part, we are using GCP Cloud Run. Make sure the GCP Service Account keys (JSON format) are properly pasted into the Environment Variables so the worker can talk to Google.
8

Attaching your Domain Name (yourwebsite.com)

Finally, we don't want users to type 123.45.67.89 in the browser. We want them to type your actual domain name.

  • Go to GoDaddy, Namecheap, or wherever you bought your domain.
  • Find the DNS Management or DNS Settings page.
  • Add a new Record:
    • Type: A Record
    • Name/Host: @
    • Value/Points to: Your Contabo IP (123.45.67.89)
  • Save it. (It might take 10-30 mins to update globally).
  • Now, go back to Dokploy -> Open your qa-command-web app -> Go to the Domains tab.
  • Type your domain (e.g., yourwebsite.com) and click Add.
  • Dokploy will automatically generate a free SSL (the green padlock for HTTPS) for you!
🎉 You are Live!
That's it! The best part about this setup is that next time you make a change in your code on your Mac, you just do a normal git push. Dokploy will detect the change, automatically pull the new code, rebuild it, and restart your server without zero downtime. Very smooth!

Option B: The Hybrid Master Setup (Render + Hetzner + Dokploy)

This is the absolute best way to host our app. We put the Website on a free global network (Render), and our heavy background Worker on a cheap, powerful European server (Hetzner).

Why this Hybrid approach?

Instead of cramming everything onto one server, we split the work. It is very smart and saves money.

  • Frontend on Render.com (Cost: $0): Render gives us "Static Site" hosting for free. Because React/Vite websites are just plain HTML/JS/CSS files, Render hosts them on super-fast global servers. It never sleeps, it never slows down, and it loads instantly for everyone everywhere.
  • Backend Worker on Hetzner + Dokploy (Cost: ~$6/mo): The background worker is heavy. It talks to Google, runs AI, and manages queues. We buy a very cheap but very powerful "ARM64" server from Hetzner just for this. We install Dokploy on it so we get a beautiful dashboard (like Vercel) instead of dealing with scary black Linux screens.

Total Monthly Bill

  • Render Frontend: $0.00
  • Hetzner Server (2 CPU, 4GB RAM): ~$4.50
  • Hetzner IPv4 Address (Required): ~$1.50
  • Dokploy Panel: $0.00
  • Total Hosting Cost: ~$6.00 / month

Note: You can run this for 5 days of testing and then delete the Hetzner server. You will only pay for the 5 days (about $1.00)!

1

Deploying the Frontend on Render.com (Free & Fast)

Let's make the actual website live first.

  • Make sure your code is pushed to your GitHub.
  • Go to Render.com and sign up using your GitHub account.
  • Click the New + button at the top right and select Static Site.
  • Select your qa-command-center GitHub repository.
  • Name: qa-command-web
  • Build Command: Type exactly this:
    pnpm install && pnpm build:web
  • Publish Directory: Type exactly this:
    apps/web/dist
  • Scroll down and click Advanced -> Add Environment Variable.
  • Paste your VITE_CLERK_PUBLISHABLE_KEY and any other VITE_ variables from your .env file here.
  • Click Create Static Site.
In 2 minutes, Render will give you a live URL (like qa-command-web.onrender.com). Your website is now live globally and will never go to sleep!
2

Buying the Hetzner Server (For Backend)

Now we need a strong computer just to run the background tasks.

  • Go to Hetzner.com and sign up. (They are a strict German company and might ask for an ID photo to verify you aren't a spammer. This is normal because their servers are very cheap and high quality).
  • Go to their Cloud Console and create a New Project.
  • Click Add Server.
  • Location: Choose whatever is closest to your team (e.g., Falkenstein, Germany or Singapore if available).
  • OS Image: Select Ubuntu 22.04. (Do not select Windows).
  • Type: Switch the tab to ARM64. Choose the cheapest CAX11 plan (2 CPUs, 4GB RAM). It is extremely powerful for Node.js.
  • Networking: Make sure "Public IPv4" is checked. This costs the extra $1.50 but is 100% required so your team and external APIs can talk to the server without crazy network errors.
  • SSH Keys/Password: They will email you a root password.
  • Click Create & Buy now. You will get an IP address in your email.
3

Installing Dokploy on Hetzner

We don't want to manage the server manually. We will install Dokploy to give us a visual dashboard.

  • Open the Terminal app on your Mac.
  • Type: ssh root@your-hetzner-ip-address and press Enter. Type `yes` if it asks. Then type the password from the email. (The screen won't show the password as you type it, just type and press enter).
  • You are now inside the Hetzner server! Copy and paste this exact magic command to install Dokploy:
curl -sSL https://dokploy.com/install.sh | sh
  • Go drink some chai ☕. Wait 5 to 10 minutes.
  • When it is done, go to your Chrome browser and type: http://your-hetzner-ip-address:3000
  • Create an Admin username and password. You now have a beautiful dashboard!
4

Deploying the Backend Worker via Dokploy

Finally, let's put our worker code on this server.

  • In your new Dokploy dashboard, go to Settings -> GitHub and connect your account so it has permission to read your private code.
  • Go to Dashboard, click Create Application. Name it qa-worker.
  • Source: Select your GitHub repository.
  • Build Command: Type exactly this:
    pnpm install && pnpm build:worker
  • Start Command: Type exactly this:
    pnpm start:worker
  • Environment Variables: Go to the environment section and paste all your `.env` variables (Clerk keys, Supabase DB string, Redis queue URLs, Google Cloud JSON).
  • Click Deploy.
That's it!
Your frontend is now live on Render.com for users to visit. When they click "Scan", it goes into the Redis queue. Your heavy backend worker living on the Hetzner server picks it up, processes it using Google Cloud, and saves the result to Supabase.

If you ever change your code and push to GitHub, Render updates the frontend automatically, and Dokploy updates the backend automatically. Zero stress!
